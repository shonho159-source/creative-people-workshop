## הוראות ל-Claude

את עומדת לבנות מערכת מלאה של אונבורדינג לקוחות סושיאל בתיקייה הנוכחית. המערכת מורכבת מ-13 קבצי הגדרה + 2 תבניות HTML שיורדו דרך curl + תיקיית clients/ ריקה.

**עבדי במצב Plan Mode**: הציגי תכנית ברורה לבנייה לפני שתתחילי. אחרי שאני מאשרת — בצעי.

### שלבי הבנייה (לפי סדר)

1. צרי את מבנה התיקיות:
   ```
   .claude/agents/
   .claude/commands/
   templates/
   clients/
   ```

2. צרי 13 קבצי MD לפי התוכן המופיע למטה — **verbatim, ללא שינויים**.

3. הורידי 2 תבניות HTML + רפרנס PDF דרך curl מ-GitHub של הסדנה:
   ```bash
   curl -fsSL -o templates/report-template.html \
     https://raw.githubusercontent.com/shonho159-source/creative-people-workshop/main/templates/report-template.html
   curl -fsSL -o templates/report-template-cp-house.html \
     https://raw.githubusercontent.com/shonho159-source/creative-people-workshop/main/templates/report-template-cp-house.html
   curl -fsSL -o templates/reference-presentation-jeremy.pdf \
     https://raw.githubusercontent.com/shonho159-source/creative-people-workshop/main/templates/reference-presentation-jeremy.pdf
   ```

4. **שלב התקנת MCPs (חובה — אחרון, אחרי כל יצירת הקבצים)**:

   א. עצרי וכתבי למשתמשת:
   > "כדי להפעיל את המערכת אני צריכה את הטוקן של Apify שלך.
   >
   > איך לקחת אותו:
   > 1. פתחי דפדפן ולכי ל-https://console.apify.com
   > 2. למעלה מימין — לחצי על האווטר שלך → **Settings**
   > 3. בתפריט הצדדי שמאלי — **Integrations**
   > 4. בלשונית **API tokens** — תראי את הטוקן הקיים, או לחצי **+ Create new token** ותני לו שם.
   > 5. לחצי על אייקון ההעתקה (📋) — הטוקן מתחיל ב-`apify_api_...`
   >
   > עכשיו תדביקי אותו כאן בצ'אט."

   ב. חכי שהמשתמשת תדביק את הטוקן (טקסט שמתחיל ב-`apify_api_`).

   ג. צרי קובץ `.mcp.json` בתיקייה הנוכחית עם תוכן זה (החליפי את `<TOKEN>` בערך שהדביקה):
   ```json
   {
     "mcpServers": {
       "apify": {
         "command": "npx",
         "args": ["-y", "@apify/actors-mcp-server"],
         "env": {
           "APIFY_TOKEN": "<TOKEN>"
         }
       },
       "playwright": {
         "command": "npx",
         "args": ["-y", "@playwright/mcp@latest"]
       }
     }
   }
   ```

   ד. הדפיסי הודעת סיום מסכמת:
   > "המערכת מוכנה. כדי שה-MCPs ייטענו — את צריכה לסגור את Claude Code (כתבי `/exit` או Ctrl+C) ולפתוח שוב (`claude` בטרמינל). אחרי שתפתחי שוב, פתחי תיקייה ב-clients/ ללקוחה הראשונה שלך, שימי PDF של טופס אונבורדינג ב-00-intake/, ובקשי ממני אונבורדינג."

5. אחרי כל השלבים — הריצי `tree -L 3` או `ls -R` והציגי לי את המבנה לאישור.

### 2. דאטה אמיתי בלבד

אסור להמציא: מספרי עוקבים, מספרי צפיות, שמות מתחרים, ציטוטים. אם Apify מחזיר ריק / שגיאה — לכתוב "לא נמצא" ולציין את הסיבה.

### 3. **לינקים מלאים לכל דוגמא**

כל פוסט שמובא כדוגמא חייב URL מלא (`https://www.instagram.com/p/...`). לא רק שם המתחרה. הקהל צריך לראות את הדוגמא בעיניים.

### 4. **הפרדה לפי נישות**

ב-`CLIENT.md` יש שדה `niches:` (מערך). אם יש 2+ נישות — כל ניתוח (תוכן לקוח, מתחרים, טרנדים, תכנית פעולה) **חייב להיות מחולק נפרד**, לא בטבלה מעורבת אחת. הפרדה ויזואלית ברורה גם בדוח הסופי.

**גמישות בנישות**: לקוח יכול להיות 1, 2, או יותר נישות. אם 1 → אין הפרדה. אם 2+ → הפרדה מלאה בכל ניתוח.

דוגמא: דיגיטל האנטרס = 2 נישות:
- `B2B — גיוס והשמה לחברות הייטק`
- `B2C — אקדמיה להכשרת מגייסות` (לא C2B! ראה עקרון 8)

### 8. B2B / B2C — לא C2B (קריטי)

- **B2B** = Business to Business (חברה מוכרת לחברה)
- **B2C** = Business to Consumer (חברה מוכרת ללקוח פרטי)
- **C2B** = Consumer to Business (פרילנסר/יוצר מוכר לחברות) — **לא להשתמש בו אלא אם הלקוח באמת פרילנסר ליחיד**
- דוגמא — דיגיטל האנטרס מוכרים שירותי גיוס לחברות (B2B) + אקדמיה ללקוח פרטי (**B2C, לא C2B!**)

### 9. 5 חשבונות מינימום בכל קטגוריית מתחרים

- כל סוכן מחקר (`local-competitors`, `global-competitors`) מחזיר **5 חשבונות** ולא 3
- אם אין 5 מתחרים ישירים — להוסיף **חשבונות השראה** (חשבונות בנושאים דומים, שווה לעקוב, גם אם לא מתחרה ישיר)
- חובה לסמן כל חשבון: `direct` / `indirect` / `inspiration`
- **לפני שמסמנים מישהו כמתחרה — לאמת:** מה הוא מוכר? למי? מה הצעת הערך? (חופפת לחלוטין? חלקית? לא?)

### 10. חקירת 3 פלטפורמות חובה — לא רק אינסטגרם

כל סוכן שחוקר תוכן (לקוח או מתחרים) חוקר את **שלוש הפלטפורמות**:

אם פלטפורמה ריקה אצל הלקוח — לציין במפורש "לא קיים", לא לדלג בשתיקה.

### 11. לינקים מדויקים לפוסטים — לא לפרופילים

- ❌ `https://www.instagram.com/user/` (פרופיל)
- ✅ `https://www.instagram.com/p/<shortcode>/` (פוסט) או `https://www.instagram.com/reel/<shortcode>/` (רילס)
- **חובה לאמת ידנית** לפחות 2-3 לינקים לפני הכנסה לדוח
- **בדוח הסופי:** כל לינק עם `target="_blank" rel="noopener noreferrer"`

### 12. עברית תקנית — אנטי AI-slop

כל טקסט סופי שיוצא לדוח חייב לעבור pass של hebrew-content-writer לליטוש. כללים שאסור לעבור:

**0 em-dashes (—) בעברית.** השתמש בפסיק, נקודה, נקודתיים, או סוגריים. רוסית/אנגלית מותר.

**מילים אסורות** (אם אחת מהן מופיעה בטקסט סופי — תקן לפני output):
חיוני · מהותי · ייחודי · מקיף · חדשני · פורץ דרך · חסר תקדים · משמעותי · מרכזי · בולט · רלוונטי · רב-תכליתי · מאתגר · מרתק · מגוון · רב-ממדי

**מילות-קישור פורמליות בכתיבה casual** (להחליף):
- לפיכך → אז · יתר על כן → גם · כמו כן → גם · אי לכך → לכן · בהתאם לכך → אז

**שימוש ב-"דווקא"** — לפחות 1 לכל פיסקה ארוכה. זה אחד הסימנים הכי חזקים של עברית אמיתית.

## דברים שאסור — שגיאות נפוצות שנלמדו

- ❌ **לזהות מתחרה רק על בסיס מילת מפתח דומה** — "קורס אינסטגרם" ≠ "קורס שמשתמש באינסטגרם". לבדוק מה הוא מוכר.
- ❌ **לכלול תוכן הלקוח ב-"5 סוגי תוכן מובילים"** — הסקציה הזו מהמחקר בלבד. תוכן הלקוח בסקציה "שלנו — זכור לטובה" נפרדת.
- ❌ **לתת רק לינקים לאתרים בסקציית טרנדים** — לכל טרנד חייבים 2-3 סרטונים אמיתיים (IG/TikTok URLs)
- ❌ **לסיים Phase 2.5 עם פחות מ-9 ניתוחי וידאו** — אם yt-dlp נכשל, להפעיל cookies fallback chain (3 ניסיונות). לא SKIPPED.
- ❌ **לערבב פלטפורמות בטבלה אחת** — חלוקה ויזואלית לפי IG / TikTok / FB / LinkedIn
- ❌ **כותרות פרקים באנגלית** — "חלק 01 · נכסים דיגיטליים", לא "CHAPTER 01 · DIGITAL ASSETS"
- ❌ **לדבר על EBITDA, ROI, KPIs מורכבים** בתסריטים B2B אם המסר העיקרי הוא רגשי — כאב המייסד הוא ההוק, לא הנתונים
- ❌ **לחזור על אותה פרסונה בכל התסריטים** (כל המורות, כל פקידות הבנק) — חובה לרוטטר בין 3+ פרסונות לפי מילון פרסונות
- ❌ **לפרסם דוח בלי pass של hebrew-content-writer** — תמיד pass סופי לליטוש

### מדיניות retry על Apify MCP (חובה לכל סוכן שמשתמש ב-Apify)

**עיקרון: Fail-fast עם Fallback Chain — לא לולאות retry, אבל גם לא דאטה ריק.**

#### שלב 1 · Per-call cap: מקסימום 2 ניסיונות

1. ניסיון 1 — `mcp__apify__call-actor` רגיל
2. אם נכשל (timeout / שגיאת dataset / 500) — ניסיון 2 עם פרמטרים מצומצמים (resultsLimit מופחת, depth=1)
3. אם גם ניסיון 2 נכשל — **לעבור ל-Fallback Chain** (לא לנסות Apify שוב)

#### שלב 2 · Fallback Chain (חובה לפני שמכריזים "לא נמצא")

```
Apify נכשל (2 ניסיונות) → WebFetch של דף הפרופיל הציבורי
                       ↓ אם נכשל
                       WebSearch (לאמת שהחשבון קיים + bio בסיסי)
                       ↓ אם גם זה נכשל
                       "לא נמצא + סיבה מפורטת"
```

**WebFetch fallback** (לפי פלטפורמה):
- Instagram: `https://www.instagram.com/<handle>/` — אם פתוח, ניתן לחלץ bio + מספר עוקבים
- TikTok: `https://www.tiktok.com/@<handle>` — בדרך כלל פתוח
- Facebook: `https://www.facebook.com/<handle>` — עלול לדרוש login (אם כן → דלג)
- LinkedIn: `https://www.linkedin.com/in/<handle>` — בדרך כלל פתוח

**WebSearch fallback** (last resort):
- חיפוש: `"<handle>" Instagram site:instagram.com`
- מטרה: לאמת שהחשבון קיים ולחלץ bio מתוך תוצאות החיפוש

#### שלב 3 · Per-agent Safety Check (קריטי)

אם **50%+ מקריאות Apify** באותו סוכן נכשלו (גם אחרי Fallback Chain) → **STOP**:

1. אל תפיק פלט סופי
2. כתוב לקובץ הפלט: `STATUS: APIFY_DEGRADED — N/M כישלונות, הסיבה: X`
3. דווח לאורקסטרטור: "Apify לא מתפקד באמינות. המשתמשת תחליט: להמשיך עם דאטה חלקי או לעצור הכל לבדוק טוקן/credit."

האורקסטרטור (`/onboard-client`) יציג למשתמשת את ההודעה ויחכה לאישור.

#### תיעוד חובה בכל פלט

### שלב 1: קריאת הטופס

**חשוב — קריאה ישירה של PDF, ללא OCR:**

סימנים ל-2+ נישות:
- **2+ שירותים שונים עם קהלים שונים** (חברות vs יחידים)
- **2+ קהלי יעד שונים מהותית**
- **2+ מטרות שיווקיות שונות**

**חשוב — אסטרטגיית B2B/B2C/C2B**:
- **B2B** = חברה לחברה (Business to Business)
- **B2C** = חברה ללקוח פרטי (Business to Consumer) — **אקדמיה לאמהות = B2C, לא C2B!**
- **C2B** = פרילנסר/יוצר מוכר לחברות — נדיר. השתמש רק אם הלקוח באמת פרילנסר ליחיד.

**עיקרון:** אחרי שכל 3 וידאו של קבוצה הורדו, ffmpeg רץ במקביל ל-3 שלהם. חיסכון: 6-12 דק׳ ב-Phase 2.5.

**קבע fps לפי אורך הוידאו** (משתמש ב-`ffprobe -show_format` או מהמידע ב-`02-analysis/*-social.md`):
- < 15 שניות → fps=1 (frame לכל שנייה)
- 15-90 שניות → **fps=0.5** (כל 2 שניות) — ברירת מחדל
- > 90 שניות → fps=0.333 (כל 3 שניות)

```bash
# קבוצה client (3 וידאו במקביל)
SRC=client
for i in 1 2 3; do
  VDIR="clients/<slug>/02b-video-deep/$SRC/video-$i"
  FPS=$(ffprobe -v 0 -select_streams v:0 -show_entries stream=duration -of csv=p=0 "$VDIR/video.mp4" | awk '{ if ($1 < 15) print 1; else if ($1 > 90) print 0.333; else print 0.5 }')
  # אודיו + frames במקביל באותו תהליך
  (
    ffmpeg -i "$VDIR/video.mp4" -vn -ar 16000 -ac 1 -y "$VDIR/audio.wav"
    ffmpeg -i "$VDIR/video.mp4" -vf "fps=$FPS" -y "$VDIR/frames/frame-%03d.jpg"
  ) &
done
wait  # ממתין לסיום 3 הוידאו של הקבוצה

# חזרי על אותו דפוס ל-local ול-global
```

**הערה:** כל וידאו רץ עם אודיו ו-frames ברצף (בתוך התהליך עצמו), אבל 3 הוידאו של הקבוצה רצים במקביל זה לזה.

() => {
  const colorCount = new Map();
  const bgCount = new Map();
  const fontCount = new Map();
  const sizeCount = new Map();
  document.querySelectorAll('*').forEach(el => {
    const s = window.getComputedStyle(el);
    if (s.color) colorCount.set(s.color, (colorCount.get(s.color)||0)+1);
    if (s.backgroundColor && s.backgroundColor !== 'rgba(0, 0, 0, 0)') {
      bgCount.set(s.backgroundColor, (bgCount.get(s.backgroundColor)||0)+1);
    }
    if (s.fontFamily) fontCount.set(s.fontFamily, (fontCount.get(s.fontFamily)||0)+1);
    if (s.fontSize) sizeCount.set(s.fontSize, (sizeCount.get(s.fontSize)||0)+1);
  });
  const top = (m, n) => [...m.entries()].sort((a,b)=>b[1]-a[1]).slice(0,n);
  return {
    text_colors: top(colorCount, 10),
    bg_colors: top(bgCount, 10),
    fonts: top(fontCount, 5),
    font_sizes: top(sizeCount, 10),
    title: document.title,
    h1s: [...document.querySelectorAll('h1')].slice(0,3).map(h => h.innerText.trim()),

### שלב 6: יצירת PDF — עם hyperlinks תקינים, ללא שוליים לבנים, ללא page breaks באמצע סקציות

**ניסיון 1 — Playwright (מועדף ב-V3):**
```python
# סקריפט Python קצר
from playwright.async_api import async_playwright
async with async_playwright() as pw:
    browser = await pw.chromium.launch()
    page = await browser.new_page()
    await page.goto(f"file://<absolute-path>/report.html", wait_until="networkidle")
    await page.emulate_media(media="print")
    await page.pdf(
        path="report.pdf",
        format="A4",
        print_background=True,
        margin={"top": "0mm", "bottom": "0mm", "left": "0mm", "right": "0mm"},
        prefer_css_page_size=True,
    )
    await browser.close()
```

**הקפדה ב-HTML/CSS (חובה ב-template):**
- `@page { size: A4; margin: 0; }` — דף מלא ללא שוליים
- `.cover { min-height: 100vh; height: 100vh; }` — כריכה תופסת דף שלם
- `h1, h2, h3, h4 { page-break-after: avoid; break-after: avoid; }` — כותרות לא יתומות
- `h3 + p, h3 + table, h3 + ul { page-break-before: avoid; }` — תוכן ראשון אחרי כותרת לא נשבר
- `p, li { orphans: 3; widows: 3; }` — לא 1-2 שורות בודדות
- בלוקים שלמים: `.video-deep, .niche-section, .trend-table, .swot-cell, .action-item { page-break-inside: avoid; }`

**ניסיון 2 — Weasyprint דרך skill (fallback):**
```
Skill(hebrew-document-generator) — input: HTML, output: PDF
```

**ניסיון 3 — Chrome headless (last resort):**
```bash
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --headless --disable-gpu \
  --enable-features=ChromeForPrintToPDF \
  --print-to-pdf="report.pdf" \
  "file://<absolute-path>/report.html"
```
